# 视频检测与评分处理耗时与优化建议

## 一、为什么处理时间较长？

可能原因包括以下几类，可对照当前环境排查。

1. **视频帧数多**  
   检测是对**每一帧**做模型推理。例如 1 分钟 30fps 约 1800 帧，几分钟的视频就会到数千帧，耗时会明显增加。

2. **当前使用 CPU 推理**  
   代码中为兼容性将检测设备固定为 `device='cpu'`，CPU 推理比 GPU 慢一个数量级以上。

3. **模型规模与未量化**  
   若使用较大的 YOLO 版本（如 YOLOv8m/l/x）或未做量化，单帧推理时间会偏长。

4. **数据集与标注量**  
   数据量或标注质量不足时，往往需要更复杂或更大的模型才能达到可用效果，间接增加推理时间；数据充足时更容易用较小、较快模型达到相同效果。

5. **评分阶段先全量检测再算分**  
   「生成评分」会先对整段视频做一次完整检测，再基于检测结果算分，因此总时间 ≈ 一次完整检测 + 评分计算，检测占绝大部分时间。

### 6. hand / 面条束检测数一直为 0、只有 noodle_rope 框

- **原因**：未加载到**抻面 3 类模型**（hand / noodle_rope / noodle_bundle），或置信度阈值过高把 hand/bundle 过滤掉了。
- **处理**：  
  1. **验证并固定 3 类权重**：在项目根目录执行  
     `python scripts/verify_stretch_weights.py`  
     会列出所有 `models/stretch_detection*/weights/best.pt` 的类别数；若有 3 类模型，会把**编号最大的 3 类**复制到 `models/stretch_detection/weights/best.pt`，服务优先使用该标准路径。  
  2. 确认 `models/stretch_detection/weights/best.pt` 存在且为 3 类；若没有，可从某次 run（如 `models/stretch_detection13/weights/best.pt`）复制到该路径，或运行 `scripts/copy_latest_model.py`。  
  3. 启动服务后看终端：若出现「当前模型有 XX 类」「使用 YOLOv8 预训练模型」等提示，说明未加载到 3 类抻面模型，需按上一步放置正确 best.pt。  
  4. 默认置信度已改为 0.15；若仍无 hand/bundle，可在 `video_detection_api.py` 的 `detect_video(conf_threshold=...)` 或调用处再调低（如 0.1）试一次。

## 二、可操作的优化建议

### 1. 降帧 / 抽样检测（优先、易实现）

- 不逐帧检测，改为按固定间隔抽帧（如每 5 帧或每 10 帧检测一帧）。
- 在 `video_detection_api.py` 的 `detect_video` 循环中加 `if frame_index % step != 0: continue`（或先读帧再决定是否送入模型），可显著减少推理次数，耗时近似按比例下降。
- 注意：抽样会略降低时间分辨率，对动作非常快的片段可能漏检，可根据业务在「速度」与「覆盖」之间折中（如 step=3~5）。

### 2. 使用 GPU 推理

- 若机器有 NVIDIA GPU 且已安装对应 PyTorch/CUDA，可将 `detect_video` 中 `device='cpu'` 改为 `device='cuda'` 或 `device=0`。
- 改完后建议做一次短视频测试，确认显存够用且推理正常，通常能获得数倍到十几倍加速。

### 3. 换用更小或量化模型

- 在效果可接受前提下，优先使用 **n/s** 等小尺寸 YOLO 版本（如 YOLOv8n），或对当前权重做 **导出与量化**（如 ONNX + 动态量化 / TensorRT），在相同设备上可明显缩短单帧时间。
- 小模型或量化后若某类目标（如细小面条）效果变差，可适当增加训练数据或在该类上做数据增强后再训一版小模型。

### 4. 适当缩小输入分辨率

- 在模型允许的输入尺寸下，对视频帧做缩放（如缩到 640 或 480 宽）再送入检测，可减少计算量；注意分辨率过小可能影响小目标，需在速度和效果间权衡。

### 5. 增加数据与迭代模型

- 数据量或标注质量不足时，优先补充**关键场景、关键类别**的标注数据，再训练一版**小尺寸模型**（如 n/s），往往能在保持或提升效果的同时加快推理。
- 若当前模型已明显过参数化（如用 l/x 但数据不多），可尝试用相同数据训练 n/s 版本对比效果与速度。

### 6. 前端与体验（已实现）

- 检测与评分已支持**异步任务 + 进度轮询**，界面上有进度条与状态文案，无需再依赖终端看进度。
- 生成评分完成后会**跳转到综合评分页**（抻面 / 下面及捞面）以图表方式展示，不再仅显示占位规则 JSON。

## 三、建议的优化顺序

1. **先做降帧/抽样**：改一处循环即可，收益高、风险低。  
2. **有 GPU 则开启 GPU**：改 `device` 并验证。  
3. **再考虑小模型或量化**：在保证效果的前提下换小权重或导出量化模型。  
4. **最后根据效果决定是否补数据、重训小模型**。

按上述顺序逐步优化，一般能在不显著牺牲效果的前提下，将长视频的处理时间降到可接受范围。
